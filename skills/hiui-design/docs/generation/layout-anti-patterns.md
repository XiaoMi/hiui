# 布局反模式清单

## 这份文件负责什么

这份文件用于跨页型约束“布局看起来能渲染，但结构已经错了”的情况。

适用于所有页面类型：

- 普通表格
- 数据统计表
- 数据可视化
- 树形表格 / 左树右表
- 抽屉表单 / 全页编辑
- 抽屉详情 / 全页详情
- 异常反馈页
- 未命中现成页型的新页面

## 使用顺序

生成页面时，默认顺序是：

1. `rules/generation-rules.md`
2. 本文件
3. 命中页型后再读对应专章和 shared 文档

验收页面时：

1. `rules/validation-checklist.md`
2. 本文件
3. 再回到命中的页型专章逐项核对

## 总原则

- 布局空白问题优先按结构失败处理，不按视觉微调处理
- 页面优先自然高度收口，不为“整齐”牺牲结构正确性
- 没有书面设计依据时，不要引入跨列等高卡片、固定高卡片或额外占位容器
- 没有书面设计依据时，不要用渐变背景、渐变描边或渐变高亮去掩盖结构层次不足
- 不要用额外 `overflow`、`min-height`、空节点、假 `padding` 去掩盖布局错位

## 必须显式写出的默认值

### 多列工作区

- 父级若使用 `grid` 组织左右列、多图表列或主次分区，默认显式写 `align-items: start`
- 不要依赖浏览器默认 `stretch`

### 纵向卡片列

- 纵向卡片列若使用 `grid`，默认显式写 `align-content: start`
- 默认保持 `grid-auto-rows: auto`
- 不要把卡片列写成隐式等高容器

### 卡片内部

- 卡片若使用 `grid` / `flex` 组织 `header + body`，正文区域默认按内容自然高度收口
- 不要在无依据时给正文层写 `flex: 1`、`height: 100%`、`min-height`、`grid-template-rows: auto 1fr`
- 不要让 header 因父级等高而被动获得超出内容所需的高度
- 一级结构卡默认不要声明 `height: 100%`、`min-height: 100%`；百分比高度属于壳层 / 滚动容器策略，不属于区块卡自身职责

### 图表容器

- 图表卡片的高度由“标题节奏 + 图表固定高度 + 卡片内边距”共同决定
- 不要再额外给图表外层包一层会吞掉剩余高度的容器
- 不要为了让多个图表卡看起来一样高，去拉伸标题区或图表上方留白

### `Descriptions` / 详情块

- `Descriptions`、详情列表、时间线、反馈正文都应紧贴各自标题节奏自然收口
- 不要把 `Descriptions` 或时间线外层套进 `flex: 1` 或隐式 `1fr` 容器里
- 不要因为右侧列更高，就把左侧详情块一起拉成等高

### 主表面归属

- 一级结构卡、图表卡、时间线主卡、详情主卡默认使用白底主表面，或复用共享壳层已给出的主表面
- `#F2F4F7`、`#EDEFF2`、`#F5F8FC` 等浅灰底默认只用于外槽、二级包裹层或弱强调面
- 不要把浅灰底直接刷到一级结构卡上，再试图靠描边、内边距和大面积空白补层级

### KPI / 指标组

- 若 KPI / 指标组位于窄侧栏、`secondary`、左侧卡片列等受限宽度区域，不要只因位置就把它直接判成 `embedded metric surface`；先判断该组是否承担主扫读任务，以及是否重复性出现在卡片列表中。重复性卡片列表中的指标默认使用 `16px / 24 / 600`
- 若该 KPI / 指标组确认仍需使用 `24px / 32 / 600`，或包含金额类长值，默认不要用固定等宽三列
- 优先使用内容感知的非等宽列、`auto-fit + minmax`，或把该 KPI 组提升出窄侧栏
- 不要把“存在合法指标主值 token”误解成“布局可以忽略最小宽和列策略”

## 反模式

### 1. 父级 `grid` 默认 `stretch`

典型症状：

- 左右列被强行拉成一样高
- 某一列内容较少，但每张卡都被无意义拉长

禁止：

- 省略多列工作区的 `align-items`
- 指望默认 `stretch` 自动得到“整齐”的卡片高度

正确做法：

- 父级显式写 `align-items: start`

### 2. 卡片列继续二次拉伸

典型症状：

- 同一列里每张卡高度被均分或被连续抬高
- 上一层列等高后，下一层卡片继续被拖着变高

禁止：

- 卡片列在 `grid` 下不写 `align-content: start`
- 用整列等高去模拟视觉节奏

正确做法：

- 卡片列显式写 `align-content: start`
- 卡片高度交给内容自身决定

### 3. `header + body` 容器用隐式 `1fr`

典型症状：

- 标题下面出现大面积空白
- 图表或 `Descriptions` 被压到卡片底部
- 时间线第一条和标题之间隔出异常空带

禁止：

- `grid-template-rows: auto 1fr` 用在非固定高卡片
- 正文层无理由写 `flex: 1` / `height: 100%`

正确做法：

- 让卡片正文按内容本身高度展开
- 只有设计稿明确要求固定高卡片时才允许使用 `1fr`

### 4. 用固定高或最小高掩盖结构问题

典型症状：

- 卡片内容不多，但容器高度很大
- 同一页不同卡片在视觉上出现无意义的“空仓”
- 某个区块底部被补出一整截空白，甚至出现越过相邻区块高度的假重叠感

禁止：

- 用 `min-height: 400px`、`height: 100%`、`min-height: 100%`、`100vh` 之类补丁强行撑开卡片
- 用固定高制造虚假的整齐
- 把一级结构卡高度绑定到父层剩余高度、grid track 或百分比高度，再误以为这是“追平相邻卡片”

为什么这类问题会造成“超出相邻卡片高度”的错觉：

- `height: 100%` / `min-height: 100%` 追随的是父级可用高度或 grid track，不是相邻卡片真实内容高度
- 当父级本身被 `stretch`、显式高度、剩余空间分配或更高列的轨道尺寸抬高时，子卡片会跟着拿到比邻居更高的尺寸
- 这时即使原始意图只是“别太短”，结果也会表现成标题下方大面积空白、灰底尾巴、甚至像与下一个区块发生重叠
- 所以这类现象默认按结构高度所有权错误处理，而不是按简单“等高追平”理解

正确做法：

- 先修正父级拉伸关系，再决定是否真的需要固定高度
- 一级结构卡默认回到 `height: auto` / `min-height: auto`，让内容自身决定收口

### 5. 用额外滚动容器掩盖空白

典型症状：

- 卡片内部出现第二层滚动
- 明明内容不多，却因为固定高和滚动导致视觉松散

禁止：

- 用 `overflow-y: auto` 掩盖布局错误
- 在图表、详情、时间线外层额外包滚动壳

正确做法：

- 先保证主工作区只有一条正确滚动链
- 仅在真实长列表或真实表体滚动场景下启用内部滚动

### 6. 用 `gap` / `padding` 模拟被拉坏后的节奏

典型症状：

- 卡片结构本来已被拉坏，又继续通过加大 `gap` / `padding` 修饰
- 标题和正文之间的空白越来越大

禁止：

- 把结构性空白误判成“间距不够”
- 通过继续增大 `padding-top`、`margin-top`、`gap` 去补视觉

正确做法：

- 先修掉拉伸链路，再重新判断是否真的需要调整间距

### 7. 用渐变冒充层级

典型症状：

- 英雄区、摘要区、指标卡或优先级卡片被刷成大面积渐变
- 页面本身没有复杂品牌场景，却通过渐变制造“高级感”

禁止：

- 业务页面主容器、卡片、按钮态、状态点默认使用渐变背景
- 用渐变替代本应由留白、边框或文字层级表达的信息结构

正确做法：

- 默认回到纯色背景、纯色描边和稳定字阶
- 只有设计稿、书面规范或反馈插画资源明确要求时才允许保留渐变

### 8. 用浅灰底一级卡片冒充层级

典型症状：

- 详情主卡、时间线主卡、图表主卡整体刷成浅灰色
- 页面已经有稳定白底主体，却又在主体内部造出第二套浅灰大卡层级
- 内容一少就露出大面积灰底垫片感

禁止：

- 把 `#F2F4F7`、`#EDEFF2`、`#F5F8FC` 直接作为一级结构卡主表面
- 用“浅灰底大卡片 + 描边 + 内边距”替代本应由标题层级、分组与留白表达的信息结构

正确做法：

- 回到白底主表面和稳定字阶，让层级由结构而不是底色承担

### 9. 指标 surface 选错后又被固定等宽三等分

典型症状：

- 左侧卡片列或 `secondary` 里的金额指标被裁切、溢出或把卡片撑坏
- 实现侧把问题归因成“指标数字太大”，但真正的结构问题是既没有先判断 `metric surface`，又把受限宽度里的 KPI 组写成了固定等宽三列
- 明明只有金额列更长，却仍被强制和另外两个短数字用同一列宽

禁止：

- 在窄侧栏、`secondary`、左侧卡片列中，不经 `metric surface` 判断就把 KPI 组默认套成 `24px / 32 / 600`
- 在窄侧栏、`secondary`、左侧卡片列中，对 `24px / 32 / 600` KPI 主值或金额类长值使用 `grid-template-columns: repeat(3, minmax(0, 1fr))`
- 用“需求写了 3 个指标”作为固定三等列的理由
- 把布局问题直接降格成“随便把字号调小”

正确做法：

- 先判断该组的业务角色与是否重复性出现在卡片列表中；若属于重复性卡片列表中的指标、局部补充摘要或详情头本地统计，则使用 `16px / 24 / 600`
- 使用内容感知的非等宽列，让金额 / 长值列获得更宽槽位
- 或在达到最小宽度阈值后再换行，而不是先天写死三等列
- 若该 KPI 组本质上承担总览任务且副栏过窄，优先把它提升出 `secondary`

- 一级结构卡回到白底或透明；浅灰底只留给外槽、二级包裹层或弱强调信息面

### 9. 手写 Timeline-like 结构绕开现成组件

典型症状：

- 页面里出现“竖线 + 圆点 + 文本块”的手写时间流结构
- 每一项都单独写节点、伪元素和连线，只是视觉上模仿 `Timeline`
- 时间线标题、节点间距和正文节奏在不同页面里持续漂移

禁止：

- 在 `Timeline` 语义、数据结构和交互都能承接时，仍然手写等价结构
- 以“视觉更自由”“这个页面只是详情页附属区块”为理由绕开组件复用

正确做法：

- 优先使用 `Timeline`，仅在组件语义确实不匹配且有书面依据时再考虑旁路实现

### 10. 用深层选择器魔改组件骨架层

典型症状：

- 页面虽然复用了 HiUI 组件，但又继续通过 `:global(.hi-v5-*)`、深层选择器或内部 className 覆写去改组件内部结构
- 组件外壳还在，但左右槽位、轴线位置、节点几何、内容分布已经变成另一套排版
- 视觉上出现“双节点”“错位”“内容槽和轴线分离”“看起来像用了组件，其实已经半手写”的现象

禁止：

- 直接覆写组件内部承担结构职责的核心类名和几何规则
- 把组件公开骨架槽位继续当成业务页自定义排版容器使用
- 在 `Timeline` 这类结构组件上，继续改写左右槽位、圆点、轴线、图标定位
- 同样不要在 `PageHeader`、`QueryFilter`、`Table`、`Descriptions`、`Empty`、`Tag`、`Progress`、`Steps` 这些常用组件上继续改写内部结构节点

正确做法：

- 若已有组件语义正确，就接受其骨架结构，只替换数据与业务内容
- 若组件骨架与目标设计确实冲突，先判断组件选型是否错误，再决定是否申请书面豁免

## 按页型提醒

### 表格类页面

- 不要因为右侧表格更高，就把左侧统计卡、树区或筛选卡一起拉高
- 不要为了让分页吸底而给整张卡片加无依据高度

### 数据可视化页

- 不要让不同图表卡片为了等高互相拉伸
- 不要让标题区、说明区承担被动占位职责

### 编辑页

- 不要把字段分组容器拉成等高卡片
- 不要用 `grid` 纵向间距掩盖字段自身节奏错误

### 详情页

- 不要因为右侧经营看板更高，连带把左侧 `Descriptions` 或关键触点卡片拉高
- 不要把 `summary`、`Descriptions`、时间线、图表的空白误修成更大的内边距

### 反馈页

- 不要为了撑满白底面板而给反馈图示或按钮区额外加占位高度
- 居中反馈应是内容居中，不是靠空白堆出来

## 快速自检

- 卡片标题与正文之间是否存在明显空带
- `Descriptions` 下方空白是否明显超过字段真实高度
- 图表上方是否被额外拉出大块留白
- 时间线、列表块、反馈正文是否被均匀拉高
- 多列页面里，内容少的一列是否被迫和内容多的一列等高
- 当前修复动作是否只是继续加 `padding / gap / min-height / overflow`

只要以上任一项回答为“是”，默认先回到布局结构排查，不要直接做视觉微调。
