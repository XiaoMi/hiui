# UX Walkthrough 执行 SOP

这份文件定义 `ux-walkthrough` 的正式执行顺序。

## 核心原则

- 先确认能不能做，再进入正式走查
- 证据不足时先补证据，不提前下结论
- 完整报告和本地 `.docx` 都完成后，任务才算完成

## 默认入口

默认只走两个高层入口：

1. `scripts/precheck_walkthrough.py`
2. `scripts/generate_docx.py`

只有在需要定位单层问题时，才拆到底层脚本。

## 八个阶段

1. 阶段 0：输入接收与可执行性判断
2. 阶段 1：模式判定
3. 阶段 2：证据就绪判断
4. 阶段 3：证据采集
5. 阶段 4：问题判断与分级
6. 阶段 5：正式报告生成
7. 阶段 6：本地 `.docx` 生成
8. 阶段 7：最终完成判断

## 阶段 0：输入接收与可执行性判断

- 先运行 `scripts/precheck_walkthrough.py <输入> ... --json`
- 优先查看 `mode`、`status`、`next_action`
- 通过条件：输入类型明确，且当前输入具备继续推进的基础条件

不能继续时按下列方式处理：

- 无法判断输入类型：只问一句最小必要澄清
- URL 仍停留在登录页：暂停正式走查，等待用户登录
- 截图零散且不足以形成完整结论：明确标记“证据不足，暂不进入正式交付”
- 代码仓库已被结构化识别为非页面项目：不继续正式走查，直接明确告知当前输入不是可执行的页面项目，并提示用户补充真正的页面输入

## 阶段 1：模式判定

- 若已经执行 `precheck_walkthrough.py`，优先复用它返回的 `mode`
- 只有在单层调试时，才单独看 `scripts/detect_input_mode.py`

可选模式只有三种：

- `code`
- `url`
- `screenshot`
- 通过条件：当前任务已有唯一主模式，且已读取对应模式说明文件

无法直接推进时按下列方式处理：

- URL 因登录态缺失无法继续，但用户能提供登录后截图：显式降级为 `screenshot`
- 代码模式缺少局部上下文，但仍能扫描整体结构：继续保留 `code`

## 阶段 2：证据就绪判断

- 若已走统一前置检查，优先复用 `scripts/precheck_walkthrough.py` 返回的 `gate` 结果
- 若只单独调试证据门禁，再运行 `scripts/check_evidence_gate.py --source <mode> ... --json`
- 先看结构化结果里的 `status`、`next_action`、`should_continue`

### `code`

- 项目目录可读
- 能运行扫描脚本，或能手动识别核心页面与结构

### `url`

- 已进入真实业务页
- 可以进行截图留证

### `screenshot`

- 用户已提供可读截图
- 截图数量和顺序足以支持本次结论

证据不足时的处理方式：

- 证据不足时，只能输出“待补充信息”或“待交互验证”，不能进入正式交付
- 若 `check_evidence_gate.py` 返回 `status=needs_input`，应优先补证据而不是直接结束
- 若 `check_evidence_gate.py` 已给出固定失败出口，再直接明确说明当前为什么无法形成完整交付

## 阶段 3：证据采集

### `code`

- 页面结构扫描结果或手动扫描结果
- 问题涉及的页面、文件、组件、函数或状态定位

### `url`

- 首屏截图
- 关键交互后截图
- 必要时的问题标注图

### `screenshot`

- 原始截图路径
- 必要时的局部确认图
- 必要时的问题标注图

不能继续时先回到补证据：

- 截图不够：先补截图，再继续
- 标注不准：先回到对象确认，再重新标注

不允许：

- 不要先写问题，再回头硬找图凑证据
- 不要用模糊红框代替真实问题定位

## 阶段 4：问题判断与分级

固定顺序如下：

1. 先结合 `ux-checklist` 和 `issue-examples` 形成候选问题
2. 再过 `ignore-list`，去掉不该报或证据不足的问题
3. 最后过 `severity-rubric`，为保留下来的问题定优先级

- 通过条件：所有问题都经过优先级判断，证据不足的问题已标为“待确认项”或“待交互验证”

不允许：

- 不要凭感觉直接写 P0/P1/P2
- 不要把纯猜测写成确定问题

## 阶段 5：正式报告生成

- 通过条件：对话中已有完整报告，而不是摘要；用户可以直接看到完整问题列表、建议与定位信息

不允许：

- 不要只给摘要而不输出完整正文
- 不要只给图片路径而不让用户看到问题内容

## 阶段 6：本地 `.docx` 生成

标准入口如下：

- 先整理结构化 `report.json`
- 再调用：

```bash
python3 {SKILL_DIR}/scripts/generate_docx.py --report-json <report.json> --output <output.docx> --json
```

- `report.json` 字段约定以 `references/report-json.md` 为准
- 生成后的文件命名、嵌图与无图规则以 `references/report-docx.md` 为准

- 通过条件：本地 `.docx` 已生成，且文件名和交付内容可直接使用

生成失败时：

- `.docx` 未生成成功：不能把任务说成完整完成；应直接说明卡住位置和直接原因

## 阶段 7：最终完成判断

以下条件必须同时满足，任务才算真正完成：

1. 已完成模式判定
2. 已满足证据门禁
3. 已完成正式问题判断与分级
4. 已在对话中输出完整报告
5. 已生成本地 `.docx`

换句话说：

- 已分析，不等于完成
- 已出报告，不等于完成
- 只有阶段 0 到阶段 6 全部走通，才算完成
