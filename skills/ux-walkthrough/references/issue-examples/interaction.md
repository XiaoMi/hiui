## disabled-button-tooltip

### [P1] 禁用按钮无解释 — 用户不知为何不可操作

**checklist**：UX-1.4  
**报告分类**：交互问题

**位置**：按钮组件

**问题描述**：Disabled 仅降透明度，Hover 无提示。

**设计理由**：用户无法判断是未填完还是系统故障。

**改进方案**：
- opacity 0.4 + Hover Tooltip「请先填写手机号」

## interactive-surface-missed

### [P1] 交互面不可发现或未验证 — 用户不知道这里能点击/滑动/长按

**checklist**：UX-1.4、UX-2.4  
**报告分类**：交互问题

**位置**：图标按钮 / 卡片 / 列表项 / 看板列 / 横向滚动区 / 长按或 hover 才出现的操作

**问题描述**：界面存在可点击、可滑动、可拖拽、可长按、可右键或 hover 后才出现的交互，但默认态没有明显线索，触发后也缺少反馈或替代路径。用户可能不知道功能存在，走查时也容易只验证主路径而漏掉这些操作。

**设计理由**：隐藏交互会降低功能可发现性，尤其是状态切换、快捷操作、批量处理、横向看板、移动端手势等场景；如果没有提示和反馈，用户会把功能当成不可用或误以为系统无响应。

**改进方案**：
- 对所有可交互元素提供 hover/focus/pressed/disabled 等状态反馈
- 横向滑动、拖拽、长按、右键等低可见交互提供显性提示或替代入口
- 对列表项/卡片/表格行至少验证默认态、hover 态、选中态、快捷操作和操作后状态

## loading-missing

### [P1] 加载/反馈缺失 — 提交表单无状态

**checklist**：UX-1.5、UX-1.6  
**报告分类**：交互问题

**位置**：`src/components/OrderForm.jsx` → 提交按钮

**问题描述**：点击「提交订单」后无 loading、无遮罩、无结果提示，用户可能重复点击。

**设计理由**：系统状态可见性——操作后必须在合理时间内给反馈。

## false-success-feedback

### [P0] 结果反馈失真 — 页面提示成功但实际失败

**checklist**：UX-1.6、UX-2.3  
**报告分类**：交互问题

**位置**：保存 / 提交 / 审批操作结果反馈

**问题描述**：用户点击提交后页面提示「操作成功」，但列表状态未更新或后端实际返回失败，用户以为任务已完成。

**设计理由**：系统反馈必须与真实结果一致；假成功会导致用户漏处理、业务数据错误或重复沟通。

**改进方案**：
- 只有服务端确认成功后再展示成功反馈
- 失败时展示原因、影响范围和下一步处理方式

## partial-success-no-detail

### [P1] 部分成功无明细 — 批量操作结果不可追踪

**checklist**：UX-1.5、UX-1.6  
**报告分类**：交互问题

**位置**：批量导入 / 批量审批 / 批量删除结果页

**问题描述**：批量处理完成后只提示「处理完成」，没有区分成功数、失败数和失败原因，用户不知道哪些数据还需要补处理。

**设计理由**：批量任务天然存在部分成功场景；没有明细会把系统成本转嫁给用户人工核对。

**改进方案**：
- 展示成功数、失败数、处理中数量
- 提供失败明细下载、重试失败项或跳转到失败列表

## submit-failure-loses-input

### [P0] 提交失败丢输入 — 用户无法恢复任务

**checklist**：UX-2.2  
**报告分类**：交互问题

**位置**：表单提交 / 弹窗确认 / 多步骤向导

**问题描述**：用户提交失败后弹窗关闭或页面刷新，已填写内容全部丢失，且没有草稿、重试或返回修改入口。

**设计理由**：失败后不可恢复会把一次系统异常变成任务阻断；复杂表单和多步骤流程尤其容易造成高成本返工。

**改进方案**：
- 保留当前输入和步骤位置
- 在原位置展示失败原因，并提供「重试」「返回修改」「保存草稿」

## delete-no-recovery

### [P0] 流程死胡同 — 删除后无法恢复

**checklist**：UX-2.2  
**报告分类**：交互问题

**位置**：`src/views/UserList.vue` → 批量删除

**问题描述**：批量删除后数据直接消失，无确认、无回收站、无撤销。

**设计理由**：不可逆操作需确认与恢复路径；批量误删成本极高。

**改进方案**：
- 二次确认（含数量）→ 删除后 toast + 5s 撤销入口

## ambiguous-context-action-icon

### [P1] 上下文操作图标语义误导 — 用户可能把状态切换理解成删除/关闭

**checklist**：UX-2.2、UX-2.5  
**报告分类**：交互问题

**位置**：列表项 / 卡片 / 表格行 / 树节点 / 看板卡的 hover 快捷操作

**问题描述**：某个上下文操作只用图标表达，且图标的通用语义与真实动作不一致。例如实际动作是取消置顶、取消收藏、移出分组、断开关联、恢复默认状态，但使用 `X`、垃圾桶、关闭、减号等容易被理解为删除/关闭的图标。

**设计理由**：隐藏快捷操作通常缺少上下文解释，用户会优先按图标通用语义理解。语义误导会让用户不敢点击，或误以为会删除对象、关闭任务、移除数据，造成操作成本和误操作风险。

**改进方案**：
- 状态切换类操作使用成对图标与文案，例如「置顶 / 取消置顶」「收藏 / 取消收藏」
- hover 时展示明确 Tooltip，说明真实动作和影响范围
- 高风险动作使用二次确认或撤销路径；非破坏性状态切换避免使用垃圾桶、`X` 等破坏性/关闭语义图标

## permission-entry-mismatch

### [P1] 权限入口错配 — 无权限用户看到可操作入口

**checklist**：UX-1.4、UX-3.2  
**报告分类**：交互问题

**位置**：列表操作列 / 详情页按钮 / 菜单入口

**问题描述**：普通用户可以看到「审批」「删除」「导出敏感数据」等入口，点击后才提示无权限。

**设计理由**：权限体验应在入口层就与角色能力一致；点击后才失败会制造误导，也可能暴露敏感能力存在。

**改进方案**：
- 无权限入口隐藏或禁用，并在必要时说明原因
- 提供「申请权限」或联系管理员路径

## modal-close-methods

### [P2] 弹窗关闭方式单一 — 仅支持点击 X

**checklist**：UX-2.4  
**报告分类**：交互问题

**位置**：模态弹窗

**问题描述**：弹窗仅 X 关闭，不支持 Esc 和蒙层点击。

**设计理由**：键盘与鼠标用户习惯不同；应支持多种关闭方式。

**改进方案**：
- X + Esc + 蒙层点击（业务弹窗可禁止蒙层关闭）

## keyboard-inaccessible

### [P2] 键盘不可操作 — 仅支持鼠标交互

**checklist**：UX-2.4、UX-3.5  
**报告分类**：交互问题

**位置**：自定义下拉、树选、日期选择器等

**问题描述**：不支持 Tab、Enter、Esc、方向键，完全依赖鼠标。

**设计理由**：WCAG 基本要求；也是高效用户常用方式。

**改进方案**：
- Tab 进入/退出、Enter 选择、Esc 关闭、↑↓ 导航
- focus：outline 2px solid #1677ff
