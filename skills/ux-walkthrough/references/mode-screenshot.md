# 截图走查

## 适用场景

当用户提供的是页面截图、设计稿截图、竞品截图，或 URL 模式下暂时无法获得真实登录态但用户可以提供登录后页面截图时，使用本模式。

## 本模式门禁

- 优先运行 `scripts/precheck_walkthrough.py --image-path <截图路径> ... --json`
- 优先读取 `precheck_walkthrough.py` 返回的 `next_action`
- 必须先判断截图数量、顺序和可读性
- 原图路径必须保留
- 若截图不足以支撑完整结论，不进入完整交付

首选命令：

```bash
python3 {SKILL_DIR}/scripts/precheck_walkthrough.py --image-path <截图路径> --image-path <截图路径> --minimum-images 1 --evidence-complete auto --json
```

若只需要单独调试证据门禁，再使用底层脚本：

```bash
python3 {SKILL_DIR}/scripts/check_evidence_gate.py --source screenshot --image-path <截图路径> --image-path <截图路径> --minimum-images 1 --evidence-complete auto --json
```

若脚本输出：

- `status=needs_input`
- `reason=evidence_insufficient`

则应先引导用户补充截图，再继续推进走查。

## 证据采集

- 若问题只落在单个小控件上，必须先从原图裁出局部图确认，再回到原图标注
- 不要在整张全页图上给很小的按钮、图标或表格末列入口直接套一个模糊小框
- 若当前截图经过缩放、拼接或转存，标注前先确认最终交付使用的是哪一张图，不要混用不同版本的位置

若需生成标注图，遵循 `annotation-style.md`。

## 问题判断重点

以下仅为截图走查的额外关注点。正式问题判断时，仍需结合 `ux-checklist.md` 与 `issue-examples.md` 执行。

- 页面类型：列表、表单、详情、仪表盘、弹窗等
- 信息层级是否清楚
- 主操作是否容易被发现
- 导航关系是否明确
- 文案是否清晰
- 视觉节奏是否统一
- 是否缺少空状态、错误状态、加载状态
- 图片、Logo、图标、间距、对齐是否有明显问题

## 待确认项处理

截图模式无法直接证明以下问题：

- 点击后是否有反馈
- 流程是否真的走得通
- 错误后是否可恢复
- 悬停、键盘操作、加载过程是否正常

遇到这类问题时：

- 可以提出怀疑
- 但必须标记为“待交互验证”
- 不要把纯猜测写成确定结论

## 多张截图

若用户提供多张截图：

- 先按页面顺序分别分析
- 再补充跨页面一致性判断
- 重点关注：入口是否统一、返回路径是否一致、列表到详情的风格是否连贯、表单与提示语是否同一套规则

## 交付门禁

- 有图就引用图上的实际区域
- 没法确认交互时，不强行下定论
- 优先输出用户能直接理解的问题
- 截图是截图走查的必要输入和必要证据
- 输出的每个问题原则上都应能对应到原图或标注图
- `.docx` 的嵌图与排版方式统一按 `report-docx.md` 执行

## `report.json` 组织建议

- `source` 固定写 `screenshot`
- `scope` 建议写成：`截图数量 + 页面范围`，例如 `订单页截图 4 张`
- `issues[].location` 优先写图上可定位区域，例如“顶部筛选栏”“右侧详情面板”
- `issues[].evidence_note` 建议直接说明对应原图或标注图，例如“对应截图 2 的右上角操作区”
- `issues[].images` 原则上应直接放原图或标注图，不要只保留口头描述
- 若问题只涉及很小控件，建议在 `images` 中优先放标注后的确认图，再放原图
- 若结论属于“待交互验证”，可以保留在 `description` 或 `evidence_note` 中，但不要写成确定交互结论

## 形成一次完整 `screenshot` 交付的条件

- 截图证据完整
- 报告已输出
- 本地 `.docx` 已生成
- 若当前只是暂时截图不全、仍在等待用户补图，说明当前还不能形成完整交付
- 若已确认截图证据不足且无法继续形成完整交付，直接说明失败原因，不要停留在模糊的中间状态
- 若问题仍以“待交互验证”为主，但整体截图走查已经完整交付，仍可正常交付
