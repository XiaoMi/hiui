# URL 走查

## 本模式门禁

- 优先运行 `scripts/precheck_walkthrough.py --url <URL> ... --json`
- 优先读取 `precheck_walkthrough.py` 返回的 `next_action`
- 必须先确认是否已进入真实业务页
- 未进入真实业务页前，不进入正式 URL 走查
- 若无法获得登录态，但用户能提供登录后截图，可显式降级为截图走查

首选命令：

```bash
python3 {SKILL_DIR}/scripts/precheck_walkthrough.py --url <当前URL> --current-url <当前URL> --page-title <当前标题> --page-reachable true|false --login-required true|false --json
```

若只需要单独调试证据门禁，再使用底层脚本：

```bash
python3 {SKILL_DIR}/scripts/check_evidence_gate.py --source url --current-url <当前URL> --page-title <当前标题> --page-reachable true|false --login-required true|false --json
```

若脚本输出：

- `status=needs_input` 且 `reason=login_required`

则应等待用户完成登录，或引导用户提供登录后的页面证据。

## 登录态获取建议

内网系统的 URL 走查，优先解决“如何进入真实业务页”，不要长时间停留在登录页反复判断。

建议顺序：

1. 优先复用用户已有的浏览器登录态，而不是默认新开无状态页面
2. 若当前环境支持接管用户现有 Chrome 会话，优先直接使用用户已登录的标签页，或在同一会话中打开目标 URL
3. 若当前环境只有独立 Browser 工具，不要假定它天然继承用户登录态；这类浏览器更适合公开页面、本地页面或已不依赖登录态的验证
4. 拿到真实业务页后，再把实际 `current-url`、`page-title`、`login-required` 回填给前置检查脚本
5. 若无法稳定复用登录态，再请用户先在可复用的浏览器会话里完成登录；登录完成后继续正式 URL 走查
6. 若用户暂时无法提供登录态，但能提供登录后的页面截图，可显式降级为截图走查

## Agent 获取登录态的技术路径

若当前环境支持 Chrome 类能力，优先按下面的方式推进：

1. 优先接管用户已经登录的标签页，而不是重新打开一个全新会话
2. 若还没有目标页，可在同一浏览器会话里打开目标 URL
3. 引导用户只完成“登录到真实业务页”这一步，不要求用户自己判断后续门禁
4. 登录完成后，由 Agent 继续截图、留证和正式走查

若当前环境只有不复用用户会话的 Browser 工具：

1. 不要把它当作内网登录态的主路径
2. 优先让用户在可复用登录态的浏览器里打开目标系统
3. 实在无法进入真实业务页时，再切换为截图走查

处理原则：

- 不把“卡在登录页”当成正式 URL 走查结果
- 不要求在 skill 内展开具体注入 cookie 的实现细节；优先复用已有会话
- 只要已经拿到登录后的真实业务页，就按正式 URL 走查继续推进

若脚本输出：

- `status=failed`
- `failure_preset=url_page_unreachable`

则应直接明确说明当前为什么无法进入完整 URL 走查。

## 登录失败处理

若页面停留在登录页或无法获得登录态，不要直接基于登录页继续走查系统内部页面。

先明确提示用户：

“当前看到的是登录/注册页，无法代表登录后的真实系统体验。请先完成登录或授权，完成后告诉我，我再继续走查。”

处理原则：

- 默认等待用户完成登录后继续
- 不默认把登录页当成截图走查对象
- 若用户明确要求走查登录页本身，可以单独分析登录页体验
- 若用户暂时无法提供登录态，再建议用户提供登录后的页面截图，切换为截图走查模式

## 证据采集

- 页面截图
- 结构快照
- 关键交互探索
- 关键操作后截图
- 必要时的问题标注图

生成标注图时，遵循 `annotation-style.md`；若需要快速产出，可直接调用 `scripts/annotate.py`。

## 标注图门禁

1. 打开原始截图
2. 确认问题本体是哪一个控件或状态块
3. 目标较小时先做局部确认
4. 回到原图完成标注
5. 标注完成后复核再写入最终交付

未经过这几步，不应提交最终标注图。

## 问题判断重点

以下仅为 URL 走查的额外关注点。正式问题判断时，仍需结合 `ux-checklist.md` 与 `issue-examples.md` 执行。

- 首屏层级
- 主操作是否清晰
- 导航是否可回退
- 状态反馈是否充分
- 文案是否面向业务

## 交付门禁

- 截图留证是 URL 走查的必要步骤
- 若没有完成截图留证，不应提交完整报告
- 输出的每个问题原则上都应有对应截图或标注图
- `.docx` 的嵌图与排版方式统一按 `report-docx.md` 执行

## `report.json` 组织建议

- `source` 固定写 `url`
- `scope` 建议写成：`页面 URL + 关键页面范围`，例如 `https://.../order/list + 列表页 / 详情页`
- `issues[].location` 优先写页面区域与交互位置，例如“列表筛选区”“弹窗底部操作区”
- `issues[].evidence_note` 建议写明截图对应状态，例如“点击保存前”“提交失败提示出现后”
- `issues[].images` 原则上应带真实页面截图或标注图；不要只写路径但不保留文件
- 若某问题跨多个状态才成立，可在同一问题下放多张图，并给每张图写 caption
- 若当前只拿到登录页、授权页或无留证截图，不要硬凑 `report.json` 走完整交付，应先回到门禁判断

## 形成一次完整 `url` 交付的条件

- 已进入真实业务页
- 已完成必要截图留证
- 已输出完整报告
- 已生成本地 `.docx`
- 若只是暂时停留在登录页、授权页或待用户登录阶段，且仍在等待用户继续操作，说明当前还不能形成完整交付
- 若已确认因登录态、授权态或页面不可达而无法进入真实业务页，直接说明失败原因，不要停留在模糊的中间状态
- 若没有完成截图留证，即使已经做了部分分析，也不应视为完整 URL 交付
- 若任务最终降级成截图走查，应按截图模式组织交付
