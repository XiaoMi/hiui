# 本地 DOCX 报告生成要求

## 作用

每次完成体验走查时，除了对话中的完整报告，还必须同步生成一份本地 `.docx` 报告。

## 标准入口

统一调用：

```bash
python3 {SKILL_DIR}/scripts/generate_docx.py --report-json <report.json> --output <output.docx> --json
```

- `report.json` 字段约定见 `report-json.md`
- `.docx` 的命名、内容结构和嵌图方式都以本文为准

## 文件命名

建议使用以下命名方式：

`{项目名或页面名}-ux-report-{YYYYMMDD}.docx`

若同一天生成多份，可追加模式或时间：

`{项目名}-url-ux-report-{YYYYMMDD-HHMM}.docx`

## 建议结构

文档建议包含以下内容：

1. 标题
2. 走查时间
3. 走查模式
4. 走查范围
5. 整体评价
6. 核心问题
7. 改进方向
8. 问题详情
9. 问题分布
10. 附录

## 图片规则

- 若存在问题对应截图，文档中应直接嵌入截图，而不是只写图片路径
- 若存在红框标注版截图，优先嵌入标注版
- 若使用标注版截图，优先采用符合 `annotation-style.md` 的统一标注样式
- 若没有标注版但有原图，可先使用原图
- 每个问题尽量配最相关的一张图，不要把所有截图无差别堆在文档末尾
- 问题文字与对应图片应尽量相邻排布，避免读者来回跳转
- 图下方可补一句简短说明，帮助读者快速理解截图对应的问题区域

## 无截图时

若本次走查没有实际截图，也仍需生成 `.docx` 报告。

- 保留完整文字报告
- 在文档中明确标注“本次报告无截图证据”
- 不要为了凑图而插入无关图片或空白占位图

## 模式差异

### 代码走查

- 默认可能没有实际页面截图
- `.docx` 报告允许只有文字内容
- 重点保留页面、文件、组件、函数等定位信息

### URL 走查

- 必须保留页面截图、交互后截图和问题标注图
- `.docx` 报告应把问题文字与对应截图放在一起

### 截图走查

- 必须直接嵌入用户提供的截图或标注图
- 若为多张图，按页面顺序组织内容

## 完成边界

- `.docx` 是完整交付前的必过门禁
- 若 `.docx` 尚未生成成功，就不要把任务说成完整完成
- 若已确认卡在 `.docx` 生成阶段，应直接说明卡住位置和直接原因
