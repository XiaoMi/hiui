# 忽略问题清单

以下问题在 B 端走查中应**忽略不报**,避免误判为体验问题。

> 该文件持续补充中。

---

## 1. 信息密度过高 / 扫读压力大

**忽略原因**:B 端场景强调屏效,用户需要一屏看到更多数据以提升操作效率。信息密度高是设计意图,不是问题。

**包含但不限于**:

- 表格行间距小,呈现信息量很大
- 搜索 placeholder 信息过载(支持多字段搜索,如"搜索名称/编号/手机号")
- 看板/仪表盘数据密集
- 侧边栏/面包屑/标签页占用空间(属于必要的导航结构)

**判断标准**：若信息密度服务于“减少翻页/跳转、提升操作效率”，忽略；若信息密度过高导致关键信息被淹没、操作入口难以发现、列表列字段过多（如超过9列），则仍视为问题。

---

## 2. HiUI 标准列表页的标准结构与标准布局

**忽略原因**：HiUI 标准列表页本身有固定的结构和常见布局。只要实现没有明显异常，就不应把这些标准做法本身当成体验问题。

**包含但不限于**:

- 双层左侧导航本身
- 搜索框与多筛选项并列本身
- 右上角 `更多 / 导入 / 导出 / 新增` 一类标准操作组合本身
- 行内 `编辑 / 删除` 并列本身

**判断标准**：当页面使用的是 HiUI 标准列表页结构，且当前状态、入口文案、操作位置都清楚可识别时，上述内容应忽略不报；只有在明显偏离 HiUI 标准实现，或已经明显影响理解、效率、操作与误操作风险时，才继续作为问题提出。
