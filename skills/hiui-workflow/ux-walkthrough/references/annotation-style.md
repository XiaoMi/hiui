# 标注图规范

**优先级：定位准确 > 样式统一。** 先按 § 标注流程 拿到坐标并在原图上叠加，再套用 § 样式规则。

若本次走查需要生成问题标注图，统一使用「浅红高亮区 + 说明条 + 箭头或短引导线」。不要用粗实线大红框围死内容，也不要用大面积实心红块遮挡正文。

---

## 标注流程（P0，先于样式）{#workflow}

每张标注图必须按顺序完成以下五步；**任一步失败则不得交付错误全图标注**（见 § 定位失败降级）。

| 步骤 | 动作 | 章节 |
|------|------|------|
| 1 | 锁定问题对象 | [#step-1-lock](#step-1-lock) |
| 2 | 取得像素坐标 | [#step-2-coords](#step-2-coords) |
| 3 | 在原图叠加高亮 | [#step-3-overlay](#step-3-overlay) |
| 4 | 像素级复核 | [#step-4-verify](#step-4-verify) |
| 5 | 定位失败则降级 | [#step-5-fallback](#step-5-fallback) |

`mode.md` 中的「确认控件 → 局部确认 → 原图标注 → 复核」即对应步骤 1 → 2/局部 → 3 → 4。

### 1. 锁定问题对象 {#step-1-lock}

标注前，先明确这条问题真正指向的对象是什么。

- 单个按钮、图标、输入框、页签 → 框该控件本身
- 空状态、禁用态、筛选区 → 框对应状态块
- 标题/副标题层级问题 → 框相关文字行，不要框整个 header 模块
- **禁止**先凭整页感觉画大框，再让说明去迁就框的位置

### 2. 取得像素坐标 {#step-2-coords}

**禁止**在整张图上凭肉眼估位置；**禁止**用 AI 生图 / 整页重绘代替在原截图上叠加。

#### URL 模式（有浏览器 / 页面 inspection 能力）

1. 对**即将用于标注的同一张截图**对应的页面状态，用 DOM / accessibility 树锁定目标元素（selector、ref 等）
2. 用页面 inspection 工具取得 bbox（如 Playwright `bounding_box()`、CDP `DOM.getBoxModel`、或当前 Agent 环境提供的等效 API），得到 `(x, y, width, height)`
3. 坐标必须与截图一致：同一 viewport、同一 scroll、同一 devicePixelRatio；截图后若页面滚动或缩放，须**重新截图 + 重新取坐标**
4. 截图范围须固定（viewport 内容区）；若截图含浏览器 chrome，坐标原点须与截图左上角一致

#### 截图 / 设计稿模式（无 DOM）

**禁止**直接对 `annotate.py --box x,y,w,h` 手填坐标。须走工具链：

```bash
# 1. 限定 tight region（仅包住目标，不要框整页/整模块）
python3 <SKILL_ROOT>/scripts/locate_in_screenshot.py \\
  --input raw.png \\
  --find-text "1,2897" \\
  --region 680,55,340,48 \\
  --occurrence last \\
  --padding 8 \\
  --verify \\
  --json

# 2. 生成 3x 局部预览，肉眼确认框贴边目标
python3 <SKILL_ROOT>/scripts/preview_bbox.py \\
  --input raw.png --bbox <上一步 bbox> --padding 0 \\
  --output previews/issue-1-preview.png --json

# 3. 一键定位 + 预览 + 标注（推荐）
python3 <SKILL_ROOT>/scripts/annotate_issue.py \\
  --input raw.png \\
  --output output/annotations/issue-1-annotated.png \\
  --find-text "1,2897" \\
  --region 680,55,340,48 \\
  --occurrence last \\
  --label "金额千分位错误" \\
  --preview-dir output/annotations/previews \\
  --json
```

要点：

1. **`--region` 必须紧贴目标**（常见偏移原因：region 过大，OCR 命中同文案的其它行/列）
2. **`--verify`**：定位后在 bbox 内二次 OCR，未命中 query 则拒绝继续
3. **`preview_bbox.py`**：交付前必看 3x 局部图；框内不是目标 → 缩小 region 或换 query
4. **`report.json` 的 `images[].bbox`**：记录最终坐标，供门禁校验

无 OCR 时：先 `preview_bbox.py` 配合 **局部裁切确认** 得到 `--bbox`，仍须 preview 复核；不得跳过。

Legacy 手工流程（仅当 OCR 不可用时）：

1. 先从原图裁出含目标的**局部确认图**（建议目标占局部图 30%–70%），确认对象无误
2. 记录局部图在原图中的偏移 `(crop_x, crop_y)`，再在局部坐标系内量取目标 `(x, y, w, h)`
3. 回全图坐标：`full_x = crop_x + x`，`full_y = crop_y + y`
4. 宽页、右上角入口、表格末列、弹层内控件：**必须先完成局部确认**，再回全图标注

#### 坐标记录（建议）

交付前至少在内部保留（可写入 `evidence_note` 或临时笔记）：

```text
screenshot: <文件名>
target: <按钮文案 / selector / ref>
bbox: x=<>, y=<>, w=<>, h=<>   # 相对截图左上角，CSS 像素
```

### 3. 在原图叠加高亮 {#step-3-overlay}

- **必须**在原始截图文件上 programmatic overlay（推荐 `scripts/annotate.py`，或 Pillow 等等价实现），按 § 2 的 bbox 绘制
- 高亮区 = 目标 bbox + padding（见 § 框选范围）
- 说明条、箭头在 overlay 之后添加；箭头终点落在高亮区边缘或核心对象上
- 一张标注图尽量只表达一个核心问题

### 4. 像素级复核 {#step-4-verify}

放大到 200% 检查，须全部满足 § 定位验收清单；任一项不通过 → 回到步骤 2 重取坐标，**不得**只改说明不改框。

### 5. 定位失败降级 {#step-5-fallback}

以下情况视为**定位失败**，不得交付「看起来有标注但位置不对」的全图：

- 无法取得可靠 bbox（无 DOM、目标太小、截图模糊、弹层已关闭）
- 复核时高亮未贴边目标控件
- URL 截图与当前页面 scroll / viewport 不一致

**降级交付（任选其一，按优先级）：**

> 注意：降级产物不等于通过 `url / screenshot` 的确定 issue 标注门禁。若问题要作为确定 issue 写入 `report.json`，仍须提供标注版截图与 `images[].bbox`；无法取得可靠 bbox 时，应标为「待交互验证 / 待确认」，或回到证据采集补图后再进入正式交付。

1. **局部裁切图** + 文字说明目标（caption 写清「局部裁切：xxx 按钮」）— 优先
2. 原图不标注，仅在 `evidence_note` 写清图上位置描述
3. 标为「待交互验证」，不硬写确定问题

## 截图与坐标一致性 {#consistency}

| 风险 | 要求 |
|------|------|
| 滚动位置 | 标注所用 bbox 必须来自**同一次截图**时的页面状态 |
| devicePixelRatio | 浏览器取坐标与截图须同一 DPR；禁止把逻辑像素坐标直接套到 2x 截图 |
| 截图裁剪 | 若截图仅为 viewport，坐标原点为截图左上角 (0,0)，不含浏览器 UI |
| sticky / fixed | 取坐标时元素须已在截图中可见；勿按「应在的位置」估框 |
| 表格 / 弹层 | 先确认行列、弹层打开状态与截图一致，再取坐标 |

## 框选范围 {#bbox-sizing}

高亮区遵循「最小必要范围 + 可辨认」：

| 规则 | 说明 |
|------|------|
| padding | 在目标 bbox 四周各扩 **4–8px**（小图标可 4px，整块区域可 8px） |
| 单控件 | 能框控件就不要框整行；能框一行就不要框整个模块 |
| 多控件 | 同一问题的多个并列控件 → **分别框选**，不用一个大框连空白一起包 |
| 最小尺寸 | 高亮区任一边 ≥ **24px**；过小则扩 padding，仍须贴边目标 |
| 文字问题 | 只框相关标题/副标题/字段行，不框整卡/整页 |

判断标准：读者**只看高亮区、不看说明**，也应大致知道是哪一个对象有问题。

## 样式规则 {#style}

- 问题区域：浅红半透明高亮，保留原页面可读性
- 说明条：红色文字，浅红半透明背景，圆角 4px；一条说明只表达一个问题
- 说明文案：直接描述用户会遇到的问题，不要只写抽象标签；优先单行，最多两行
- 说明条与问题区域：原则上用箭头或短引导线连接；说明条已紧邻且无歧义时可省略箭头
- 箭头终点：落在高亮区边缘或核心对象上，不要插入正文中间
- 整体克制统一，像「点评说明」，不像「重涂页面」

### 建议参数

- 高亮区：浅红半透明，透明度 **15%–25%**（复核时可临时加 1px 描边辅助对齐，交付前可去掉）
- 箭头 / 引导线：细线、短路径、避免绕行

### 禁止项

- 粗实线大红框围死内容，或大面积实心红块遮挡正文
- 凭肉眼在整图上估位置（见 § 2）
- AI 生图、整页重绘、手动画图代替坐标 overlay
- 为编号默认加角标/小数字，除非用户明确要求
- 混用多套颜色；整张图统一红色标注语言
- 说明条过长；说明与框选对象不一致

## 说明与框一一对应 {#copy-match}

说明里提到什么，高亮区就必须真实框到什么。

- 文案写「提交按钮」→ 必须框到该按钮，不是整表单
- 文案写「手机号输入框」→ 必须框到该输入框
- 箭头终点必须落在对应高亮区上

若说明描述 A、高亮主要覆盖 B → **标注失败**，须重画或走 § 5 降级。

## 定位验收清单（交付前必过）{#acceptance}

每张标注图交付前，**全部勾选**；任一项否 → 不得作为该问题的标注证据。

- [ ] 高亮区覆盖的是**问题本体**，不是父级模块或相邻控件
- [ ] 高亮区未纳入大量无关空白/无关列/无关模块
- [ ] 说明文字中的对象名，能在高亮区内找到对应 UI
- [ ] 箭头（若有）终点落在本问题的高亮区上
- [ ] 放大 200% 看，高亮边缘与目标控件基本贴齐（padding 内，不漂到隔壁按钮）
- [ ] 标注基于**同一张原图**、同一 scroll/viewport 状态
- [ ] 未使用 AI 重绘；是在原截图上 overlay
- [ ] 读者不看报告正文，仅看此图也能理解问题大致位置

样式层面额外检查：

- [ ] 原页面核心内容仍清晰可读
- [ ] 标注不比页面内容更抢眼；若抢眼则减弱透明度或缩小高亮区

## 与 report.json 的衔接

- `issues[].images`：优先 `{ path, caption }` 指向**标注版**；若同时保留原图，标注版放在前
- `caption` 建议格式：`图 N：<问题简述>（标注：xxx 区域）`
- 若走 § 5 降级为局部裁切：`caption` 须写明「局部裁切，非全页标注」
- 无法用像素级标注时：不要伪造标注图；用原图 + 清晰的 `location` / `evidence_note`
