## main-cta-hidden

### [P0] 主操作不可见 — 核心 CTA 被折叠

**checklist**：UX-1.1、UX-1.4  
**报告分类**：流程问题

**位置**：`https://example.com/dashboard` → 页面右上角

**问题描述**：首页核心操作「新建工单」被收进「更多」下拉菜单，首屏不可见，需额外一次点击。

**设计理由**：Fitts 定律——核心操作应始终可见，不应隐藏在次级菜单。

**改进方案**：
- After：「新建工单」作为主按钮直接展示在页面右上角
- 样式：height 40px，padding 0 24px，background #1677ff，border-radius 8px

## menu-too-deep

### [P0] 导航层级过深 — 菜单嵌套超 3 级

**checklist**：UX-1.1  
**报告分类**：流程问题

**位置**：侧边导航系统

**问题描述**：菜单嵌套 4–5 级，多次点击才能触达目标页，难以返回上层。

**设计理由**：信息架构深度影响可达性；UX 体验规则建议最多 3 级菜单。

**改进方案**：
- 合并低频四级菜单到三级，或全局搜索直达
- 一级菜单 ≤9 个，二三级 ≤16 个

## steps-redundant

### [P1] 操作步骤冗余 — 修改个人信息需 5 步

**checklist**：UX-1.1  
**报告分类**：流程问题

**位置**：`src/pages/profile/Edit.tsx`

**问题描述**：改头像需：编辑 → 选图 → 裁剪 → 确认 → 保存（5 步），「确认裁剪」与「保存」可合并。

**设计理由**：Hick 定律——高频操作应缩短路径。

**改进方案**：
- After：点击头像 → 裁剪 → 保存（3 步）

## task-chain-dead-end

### [P1] 任务链路断点 — 完成后没有结果与下一步

**checklist**：UX-1.1、UX-1.3  
**报告分类**：流程问题

**位置**：创建 / 提交类流程的完成页或提交后状态

**问题描述**：用户提交申请后页面直接关闭或停留在原表单，没有明确告知是否成功，也没有查看详情、继续创建或返回列表的下一步入口。

**设计理由**：核心任务必须形成「入口 → 操作 → 结果 → 后续处理」闭环；缺少结果与下一步会让用户反复确认或重复提交。

**改进方案**：
- 提交成功后展示结果反馈，并提供「查看详情」「继续创建」「返回列表」
- 若结果需要异步处理，展示处理状态与预计完成方式

## task-lifecycle-not-verified

### [P1] 任务闭环未跑通 — 只验证入口没有验证结果

**checklist**：UX-1.1、UX-1.3  
**报告分类**：流程问题

**位置**：新建 / 编辑 / 删除 / 邀请 / 发布 / 导入等对象生命周期流程

**问题描述**：页面提供了创建、编辑、删除、邀请或发布入口，但走查只验证按钮可点击、弹窗可打开、字段可输入，没有继续验证提交结果、成功/失败反馈、列表刷新、详情可达、状态变化和删除/撤销/归档后的回收结果。用户实际使用时可能在创建成功后找不到对象、详情打不开、删除后列表仍保留旧状态，或需要反复刷新确认。

**设计理由**：对象类业务的体验质量取决于完整生命周期，而不是单个控件是否可点击。完整闭环应覆盖「入口 → 填字段 → 提交 → 结果反馈 → 列表/详情落点 → 后续编辑或清理」。缺少任一关键验证，都会掩盖真实的流程断点。

**改进方案**：
- 使用测试对象跑通创建闭环：提交后验证成功反馈、列表新增、详情可进入
- 验证后续动作：编辑后字段更新，删除/归档/撤销后列表、详情、计数和状态同步变化
- 对无法安全执行的真实删除、邀请、发布等动作，在报告附录标记「待交互验证」并说明阻塞原因，不把流程视为已跑通

## rejected-no-resubmit

### [P0] 状态恢复缺失 — 驳回后无法修改重提

**checklist**：UX-1.3、UX-2.2  
**报告分类**：流程问题

**位置**：审批详情页 / 驳回状态操作区

**问题描述**：单据进入「已驳回」状态后，只展示驳回结果和原因，没有「修改后重新提交」入口，用户只能重新创建单据。

**设计理由**：驳回不是流程终点，而是需要用户修正后恢复任务；缺少恢复路径会造成重复录入和重复数据。

**改进方案**：
- 驳回状态下提供「编辑并重新提交」
- 保留原审批记录和驳回原因，重新提交后进入新的审批轮次

## cross-page-return-inconsistent

### [P1] 跨页落点不一致 — 保存后路径不可预测

**checklist**：UX-1.3、UX-2.5  
**报告分类**：流程问题

**位置**：列表页 / 详情页 / 编辑页之间的跳转链路

**问题描述**：同一业务对象在不同入口编辑后，有的返回列表、有的停留编辑页、有的跳到首页，用户难以预测保存后的落点。

**设计理由**：跨页面流程的一致性影响用户对系统的空间感和可控感；高频任务的返回路径应稳定。

**改进方案**：
- 统一编辑保存后的落点，例如返回详情页并刷新状态
- 保留来源页上下文，如列表筛选、分页和选中项

## repeated-input-after-failure

### [P1] 失败后重复录入 — 用户输入没有保留

**checklist**：UX-2.2  
**报告分类**：流程问题

**位置**：复杂表单提交失败后

**问题描述**：表单提交失败或网络超时后，已填写内容被清空，用户需要重新录入所有字段。

**设计理由**：错误恢复的第一原则是保护用户劳动成果；复杂表单输入丢失会显著增加挫败感和业务成本。

**改进方案**：
- 提交失败后保留表单内容和用户选择
- 提供「重试」「返回修改」和明确失败原因

## form-no-grouping

### [P1] 表单无分组 — 10+ 字段平铺堆砌

**checklist**：UX-1.2  
**报告分类**：流程问题

**位置**：录入表单

**问题描述**：15 个字段全部平铺，无分组、无分隔。

**设计理由**：组块化——≥10 字段应按业务逻辑分组，每组 ≤9 个。

**改进方案**：
- 分为「基本信息」「联系方式」「附加信息」三组，每组有标题
- 分组标题：font-size 14px，font-weight 600，margin-bottom 16px

## mobile-table-scroll

### [P1] 移动端适配差 — 表格横向滚动无提示

**checklist**：UX-3.5  
**报告分类**：流程问题

**位置**：`https://example.com/orders` → 订单列表

**问题描述**：10 列表格在 375px 宽度溢出需横滑，无滚动指示，用户不知右侧还有列。

**设计理由**：无提示的溢出等于信息丢失。

**改进方案**：
- 移动端仅展示 5 列核心字段，其余进展开面板
- 或 `overflow-x: auto` + 右侧渐变遮罩提示
