# 交付模式

选择能满足用户当前请求的最小模式，并同步确定 `confirmationDepth`。用户要求快速推进时，基于明确假设继续；若下游会生成页面或原型，必须先确认生成输入或取得明确授权。

默认不要把“帮我细化需求”自动理解为 `quick-refine`。只有用户明确要快速聊一轮、暂不展开时，才适合 `light` 深度；否则优先进入 `solution-only` 或 `page-inventory`。

只有当用户明确要求正式 PRD 且下一步需要生成输入，或明确要求“细化后直接进入页面生成、HiUI 生成、原型生成或 UX 验收”时，才默认优先选择 `full-prd-to-generation`。若只是判断结果可能会被下游继续使用，但用户尚未明确要 PRD 或直接进入生成，优先停在 `solution-only`、`page-inventory` 或 `product-prd`。

## quick-refine

用于早期模糊想法，或用户明确只想先聊清楚方向时。它是探索优先的轻量出口，不是弱化版正式方案。默认 `confirmationDepth=light`。

仅在以下条件同时满足时使用：

- 用户明确要求快速澄清，而不是要求完整产品方案
- 当前不会马上输出页面清单、页面提示词或交接包
- 当前需求不是 B2B / 管理后台 / 配置治理 / 流程类高规则密度场景

包含：

- 当前理解
- 产品类型假设
- 关键假设
- 2-3 个最高影响的选项式问题
- 建议下一步

不包含：

- 正式产品方案定稿
- 完整 PRD 正文
- 页面清单、页面级提示词或 HiUI handoff
- 可直接下游消费的生成输入

若仍有高影响 `remainingDebt`，可以收口为“待确认 + 建议下一步”，但不得写成已确认，也不得假装已经进入方案或 PRD 阶段。

## solution-only

用于用户在页面设计前需要产品方案、MVP 定义或 PRD 方向时。默认 `confirmationDepth=standard`；若用于正式 PRD 评审或可复用交付物，升级为 `strict`。

包含：

- 产品定位
- MVP 目标
- 用户角色
- P0/P1/P2 范围
- 核心流程
- 功能模块
- 数据对象
- 业务规则
- 成功指标
- 风险、假设、非目标和开放问题

输出前应说明确认完整度；若仍有影响产品方案的 `remainingDebt`，列为待确认或假设。

结束确认前，至少应明确或显式假设以下高影响项中的大部分：

- 目标用户与权限边界
- 核心对象与字段粒度
- 关键规则、冲突和生效方式
- 异常、审计或历史策略

若这些内容里仍有 2 项以上未定，不要因为“已经问过两轮”就结束。

## product-prd

用于用户要求正式 PRD、需求文档、评审稿、产品文档、在线协作文档或可归档材料时。默认 `confirmationDepth=strict`。

包含：

- 完整产品 PRD 正文
- 背景与目标
- 用户与场景
- 核心功能需求
- 业务规则与异常流程
- 复杂流程/状态/协作/对象关系需要时补充图示
- 角色边界复杂时补充权限矩阵；状态条件复杂时补充状态流转表
- 页面清单
- 依赖、风险、假设和待确认项
- 验收标准
- 图示、规则表、异常流程、页面清单之间的角色名/状态名/对象名保持一致；若某概念仅是异常标签、阻塞标记或说明文案，而非正式生命周期状态，需显式注明，避免与状态图冲突

不包含：

- 页面级提示词
- HiUI 交接包
- 机器计划
- `generationInputGate` 等执行控制细节

输出渠道：

- 若当前模式包含 `product-prd` 且宿主环境提供协作文档创建能力（例如飞书），优先生成在线协作文档并返回链接。
- 若当前模式包含 `product-prd` 且无协作文档工具或协作文档写入失败，必须生成 Markdown PRD 文件并返回路径。
- 若当前模式包含 `product-prd`，不得只在消息中给 PRD 摘要或正文而不产出在线文档链接 / Markdown 路径。

若正文仍依赖高影响 `remainingDebt`，文档状态必须写为“草稿 / 待确认版本 / 评审稿”，而不是“已确认”。

若需求具备以下任一复杂性，PRD 正文默认应补图，而不是只保留文字和表格：

- 存在明显的主流程分支、审批分支或回退分支
- 生命周期或状态机对理解需求至关重要
- 2 个及以上角色协同、交接或责任边界容易混淆
- 2 个及以上核心对象之间存在强依赖、归属或产出关系

补图后还应做一次一致性检查：

- 主流程图、状态图、异常流程表、业务规则表中引用的状态名不得互相漂移
- 角色协作图中的角色集合应能映射到“用户角色”章节，不能图里新增匿名角色
- 对象关系图若引入新对象，正文应至少补充该对象的业务职责或作用
- 若异常流程里提到“阻塞”“挂起”“退回待处理”等概念，需明确它是正式状态、子状态、标签还是备注字段
- 若角色边界复杂但未提供权限矩阵，需显式说明当前为何仅用文字足以表达
- 若状态条件复杂但未提供状态流转表，需显式说明当前为何仅用状态图足以表达

## page-inventory

用于用户要求页面清单、信息架构、路由规划或范围拆解时。默认 `confirmationDepth=standard`；若页面清单会被直接用于生成，升级为 `strict`。

包含：

- 方案摘要
- 场景表
- 必要的功能/规则/数据表
- 带工作面类型、状态以及关键字段/列/筛选/操作摘要的页面清单
- 追踪矩阵
- 反向覆盖说明

页面清单中的关键页面、状态、入口和出口如果仍基于假设，必须显式标记。

若页面清单会直接被下游生成消费，或目标是 B2B / 管理后台 / HiUI 页面，关键页面不得只写“概览”“列表”“详情”这类抽象块；至少要补到本页看什么、按什么筛选、展示哪些列/字段、能执行哪些操作的粒度。

若页面或工作面选择本身还依赖未确认规则，例如抽屉还是行内编辑取决于字段量、校验方式、保存策略或批量修改模式，应先继续确认，不要直接补全。

## prompt-pack

用于下一位 agent 将基于提示词生成设计、原型或代码时。默认 `confirmationDepth=strict`。

包含：

- 方案摘要
- 页面清单
- 全局生成上下文
- 每个页面/工作面一个页面级提示词，且每个提示词都要细化到字段/列/筛选/操作级
- 每个提示词的验收标准

必须输出生成输入确认块；未确认时不得将提示词包标为 `confirmed`。

在进入生成输入确认前，先判断当前输出属于哪一类：

- `draft generation-pack`：允许先输出页面骨架、模块结构或待补字段版本，用于评审和继续确认；不得进入生成输入确认块，也不得写成 `ready-for-review`。
- `consumable generation-pack`：关键页面已经满足字段/列/筛选/操作级粒度，可被下游 agent 直接消费；只有这一类才允许进入生成输入确认块。

若页面级提示词仍大量依赖对字段、规则、权限或状态的猜测，应回退到 `solution-only` 或继续确认，而不是硬写提示词。
若提示词主体仍停留在“模块名 + 一句描述”，没有给出字段清单、列清单、筛选项、控件类型、校验或操作明细，也应视为未达标。

## hiui-handoff

用于下一步是 HiUI 页面生成、`hiui-page-workflow`、页面验收或 UX 验收，且用户明确只要最小生成交接物时。默认 `confirmationDepth=strict`。

包含：

- 带 HiUI 页型建议的页面清单
- 全局生成上下文
- 需要时包含页面级提示词；凡包含提示词时，仍需满足字段/列/筛选/操作级粒度
- HiUI 交接包
- 会影响生成/验收的假设和风险

必须包含 `requirementGate`、`generationInputGate`、确认完整度、假设和风险。未确认时只允许 `ready-for-review`、`assumption-authorized` 或 `blocked`。

对于 B2B / 管理后台页面，若权限、异常、导入/批量操作、日志/审计中仍有高影响缺口，应继续确认，或明确标为 `draft generation-pack`；不要写成 `ready-for-review`。
若页面仍无法明确关键字段、筛选项、表格列或主次操作，也应继续确认，或明确标为 `draft generation-pack`，不要直接交给下游生成。

进一步约束：

- `ready-for-review` 只适用于最小可消费的 HiUI handoff，不能用于抽象版页面骨架或模块级提示词。
- 若当前 handoff 仍只有页名、surfaceSummary、actions，而缺少 `queryControls`、`tableColumns`、`formSections` 或 `detailSections` 中的必需结构，应视为 `draft generation-pack`，继续确认。

## full-prd-to-generation

用于用户同时要求正式/可复用 PRD 与下游生成输入，或明确要求“细化完成后直接进入页面生成链路”时。默认 `confirmationDepth=strict`。

包含：

- 完整产品 PRD
- 完整产品方案
- 场景、功能、规则、数据、权限和状态表
- 复杂流程/状态/协作/对象关系需要时补充图示
- 追踪矩阵
- 全局生成上下文
- 页面清单
- 页面级提示词
- 适用时包含 HiUI 交接包
- 开放问题和下一轮确认计划

输出时，PRD 正文与提示词/交接包应分开展示或分开存放，不要混成一份正文。

默认策略：

- 若用户同时需要 PRD 与页面生成输入，就两份都给。
- 若用户明确要求“一条龙交付”或明确说“细化完直接继续生成页面/HiUI/原型”，默认两份都给。
- 若用户只是在收敛需求阶段，尚未明确要求直接进入生成，优先停在 `solution-only`、`page-inventory` 或 `product-prd`。
- 只有用户明确要求“不要 PRD”“只给 handoff”“只给提示词包”时，才退回更轻模式。

若仍有高影响 `remainingDebt`，不得隐藏在正文中；必须放入开放问题、假设或下一轮确认计划。

若正文里已经出现多个会改变页面、字段、规则或验收的关键假设，应降低收口力度，改为“待确认版本”而不是“完整 PRD”。

若 generation-pack 仍未通过字段粒度校验，应明确标为 `draft generation-pack`，并暂停生成输入确认；不要因为用户要求“一条龙交付”就把评审草稿推进到 `ready-for-review`。

## 输出约束

所有交付模式都必须满足“只输出该模式允许的内容，不额外夹带相邻模式内容”。

### product-prd

- 允许：背景、目标、非目标、用户与场景、核心功能需求、业务规则、异常流程、页面清单、依赖/风险、验收标准、待确认项/下一步
- 禁止：页面级提示词、HiUI 交接包、`requirementGate`、`generationInputGate`、机器计划、执行日志

### generation-pack

- 允许：全局生成上下文、页面清单、页面级提示词、HiUI 交接包、与生成直接相关的假设和风险
- 禁止：完整 PRD 正文、与评审归档相关的文档信息、机器计划、无关背景叙述
- 若标为 `consumable generation-pack`，关键页面必须满足字段/列/筛选/操作级粒度，并能支持生成输入确认
- 若仍为 `draft generation-pack`，必须显式标记为待确认版本，且不得进入生成输入确认块

### full-prd-to-generation

- 必须拆分为 `product-prd` 与 `generation-pack` 两个独立 section，或两个独立文件
- 任一 section 都只能包含该 section 允许的内容
- 不得把 PRD 正文、页面级提示词和 HiUI 交接包混写成一份单一正文
