在修改 `hiui-refine` 后，至少用以下样例中的 4 组提示做一次前向验证。目标不是生成完整产物，而是确认交付模式、仓库预载、PRD 分层和下游门禁是否按预期触发。

## 用法

- 每次改完高影响规则后，至少抽取 2 组样例复查
- 大改交付模式、项目上下文预载、PRD 输出或下游门禁时，4 组全跑
- 若结果与“预期行为”不一致，优先修正规则或引用链，再继续扩展

## 样例 1：抽象需求，只要方案，不要 PRD

**用户提示**

```text
帮我细化一个企业知识库搜索平台的需求，先只要产品方案和 MVP 范围，不要 PRD，也先别拆页面提示词。
```

**预期行为**

- 默认选择 `solution-only`，而不是 `quick-refine`
- 输出方案摘要、MVP 范围、角色/场景/规则框架
- 不输出正式 PRD 正文
- 不输出页面级提示词或 HiUI handoff

## 样例 2：已有仓库上下文，先扫代码再提问

**用户提示**

```text
在这个项目里帮我细化“批量导入客户标签规则”的需求，先结合当前仓库已有页面、接口和类型再问我问题。
```

**预期行为**

- 触发项目上下文预载
- 先复述 `repoFindings / repoAssumptions / repoGaps`
- 若发现现有实现相关，必须发起“沿用 / 扩展 / 重做”的确认
- 不重复追问仓库中已有答案的问题

## 样例 3：要正式 PRD，但不要页面生成包

**用户提示**

```text
把这个审批流需求整理成正式 PRD，给评审用，不需要页面提示词，也不要生成交接包。
```

**预期行为**

- 选择 `product-prd`
- 按 PRD 骨架输出或落文档
- PRD 正文不混入页面级提示词、HiUI handoff 或 gate 细节
- 若仍有高影响 `remainingDebt`，文档状态应为草稿/待确认/评审稿

## 样例 4：一条龙交付，输出 PRD + generation-pack

**用户提示**

```text
把这个 B2B 商品配置后台需求从细化到页面生成输入一条龙做完，先出评审用 PRD，再给页面清单、提示词和 HiUI 交接包。
```

**预期行为**

- 选择 `full-prd-to-generation`
- 输出拆分后的 `product-prd` 与 `generation-pack`
- 若准备进入下游生成，必须出现生成输入确认块
- 未获确认时，`generationInputGate.status` 只能是 `ready-for-review`、`assumption-authorized` 或 `blocked`
- generation-pack 中的关键页面不能只写“概览/列表/详情”，而要写到字段/列/筛选/操作级

## 样例 5：用户明确抱怨提示词太抽象，要求到字段级

**用户提示**

```text
你上次给我的零售经营页提示词还是太抽象了，不要只写“经营概览、异常清单、热销商品表”，这次请细化到字段级，至少把筛选项、指标卡字段、表格列和操作写清楚。
```

**预期行为**

- 不再输出只有模块名的一层描述
- 页面清单或页面提示词中明确筛选项、指标字段、表格列、行操作/批量操作
- 若字段仍存在高影响缺口，先继续确认或显式标注为待确认，而不是假装已可直接生成

## 样例 6：只要最小 HiUI handoff，但仍要求字段级

**用户提示**

```text
不要完整页面提示词，只给我能交给 hiui-page-workflow 的最小 HiUI handoff，但列表页的筛选项、表格列、行操作还是要写清楚。
```

**预期行为**

- 选择 `hiui-handoff`，而不是自动升级成 `prompt-pack` 或 `full-prd-to-generation`
- `pages[]` 中补齐字段级结构，而不是只给 `primaryActions`
- handoff 与交付模式保持一致，不混入完整 PRD 正文

## 样例 7：用户指出提示词太抽象后，必须回退门禁

**用户提示**

```text
你刚刚已经给过我一版 generation input 了，但这些页面提示词还是太抽象，像“候选人概览”“面试记录”“跟进区”这种模块名不够，我现在不要你继续让我选 A/D，先把字段、筛选项、表格列、操作和异常规则补齐。
```

**预期行为**

- 不得继续停留在 `generationInputGate.ready-for-review`
- 立即回退到 `strict` 确认，补问规则 / 数据 / 异常中的高影响缺口
- 不得再次直接要求用户在抽象提示词基础上选择“确认并生成”或“保留假设先生成”
- 后续若重新进入 generation review，必须先把关键页面补到字段/列/筛选/操作级

## 样例 8：正式 PRD 涉及复杂流程和多角色协作时，必须补图

**用户提示**

```text
把这个跨招聘专员、面试官、用人经理、HRBP 的面试管理流程整理成正式 PRD，给评审看。流程里有预约、面试、反馈回退、录用和入职办理，最好一眼能看懂角色交接关系。
```

**预期行为**

- 选择 `product-prd` 或 `full-prd-to-generation`，取决于用户是否同时要求生成包
- PRD 正文除表格外，还应补充流程图、角色协作图，必要时补状态图或对象关系图
- 图示属于产品层信息，可写入 PRD 正文；但不得混入页面级提示词、HiUI handoff 或 gate 细节
- 图示与正文中的角色名、状态名、对象名应保持一致；若出现“阻塞”“待保留”等概念，需明确它是正式状态还是异常标签/补充字段
- 若模型没有补图，至少要显式说明当前为何可不补图；不能默认只给文字和表格
