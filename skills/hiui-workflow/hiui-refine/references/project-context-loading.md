# 项目上下文预载

当 `hiui-refine` 运行在已有仓库/工作区中，且用户需求明显与当前项目业务相关时，先执行最小化的项目上下文预载，再进入正式需求细化。

## 触发条件

满足以下大部分信号时触发：

- 用户明确提到“在这个项目里”“结合当前仓库”“参考现有页面/模块/接口”
- 需求名词与仓库中的业务对象、模块名、页面名或字段名明显同域
- 当前请求若不读取本地证据，会显著增加 `questionDebt`

以下情况可以跳过：

- 用户明确要求忽略当前实现或只从抽象产品角度讨论
- 当前问题与仓库业务明显无关
- 扫描后未发现足以影响方案的高信号证据

## 扫描预算

- 默认只读取 `2-6` 个高信号文件
- 目标是支撑“当前理解 + 下一轮高质量问题”，不是穷举仓库
- 一旦已能识别稳定命名、主要对象、页面模式和显著冲突，即停止扩展扫描

## 扫描顺序

1. `AGENTS.md`、`README`、项目级产品/技术文档
2. 路由、模块暴露配置、导航映射、页面注册点
3. 基于用户关键词搜索 `views/`、`api/`、`types/`、`mock/`、`utils/`
4. 读取最高信号的实现、类型、接口或 mock，提炼稳定语义

## 提炼结果

预载完成后，内部至少整理：

- `repoFindings`：仓库中明确存在的对象、命名、模块、接口、页面模式或约束
- `repoAssumptions`：基于仓库证据形成、但尚未被用户确认的推断
- `repoConflicts`：仓库现状与用户口述、已有材料或当前方案的冲突点
- `repoGaps`：扫描后仍缺失、且会显著影响方案的关键未知项

## 输出要求

进入正式提问前，先用简洁语言说明：

- 仓库已知信息
- 当前假设
- 仍待确认的高影响缺口

若现有实现、字段结构、页面模式或模块边界与用户目标可能相关，必须追加一个反锚定确认问题，不得默认沿用现状。默认使用如下三选一：

```markdown
### 与现有实现的关系

- A. 沿用现状（推荐仅在目标与现有方案高度一致时使用）：继续沿用当前命名、对象和页面模式，影响是复用成本最低，但会继承当前边界和限制
- B. 在现状上扩展：保留主对象或主流程，但允许新增字段、规则、页面或工作面，影响是兼顾复用与调整，但需要确认兼容边界
- C. 重做该能力：现有实现只作为参考，不默认继承其字段结构、页面模式或模块边界，影响是自由度更高，但需要补齐更多确认
```

需要继续提问时，优先围绕这些缺口，而不是重复询问仓库中已有答案的问题。

## 停止条件

出现任一情况即可停止预载并返回主流程：

- 已能回答当前需求所依赖的大部分命名、对象或页面模式问题
- 已找到足以改变提问方向的关键冲突或复用线索
- 已达到扫描预算上限
- 未找到有效证据，并已明确记录“未发现足以影响方案的项目历史知识”

## Guardrails

- 仓库证据只能作为上下文和约束来源，不能等同于用户确认
- 现有实现是候选方案，不是产品真相
- 若仓库模式与用户目标可能冲突，必须显式发起“沿用 / 扩展 / 重做”的确认，而不是默认沿用
