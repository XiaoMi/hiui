# 确认模型

用于控制需求反问深度，避免“一轮 3 问后就结束”。主原则：每轮轻量、总过程可追踪。

## confirmationDepth

| 模式 | 适用场景 | 反问策略 |
|---|---|---|
| `light` | 快速澄清、早期想法、用户只想先聊清楚 | 通常 1 轮，每轮最多 3 个问题组 |
| `standard` | 产品方案、MVP、页面清单、范围拆解 | 多轮确认，每轮最多 3 个问题组，直到核心范围和规则清楚 |
| `strict` | PRD、页面提示词、HiUI handoff、页面生成、UX 验收 | 多轮确认 + 生成输入确认块，不允许一轮后直接结束 |

默认映射：

- `quick-refine` -> `light`
- `solution-only` -> `standard`
- `page-inventory` -> `standard`
- `prompt-pack` -> `strict`
- `hiui-handoff` -> `strict`
- `full-prd-to-generation` -> `strict`

升级到 `strict`：

- 用户要求页面生成、HiUI 生成、原型生成或 UX 验收
- 用户要求正式 PRD、PRD 评审、可复用交付物
- 输出会被下游 agent 直接用于设计、原型或代码生成

显式降级到 `light` 的前提：

- 用户明确说“先快速聊一下”“只要一轮澄清”“不要展开”
- 当前任务不会马上产出产品方案、页面清单、页面提示词或交接包
- 当前输入不是 B2B / 管理后台 / 配置治理 / 审批流程这类高规则密度需求

### depthFloors

`confirmationDepth` 决定的不只是“每轮问几个问题”，还决定结束确认前至少要覆盖到什么粒度。

| 模式 | 最低深度底线 | 不满足时的动作 |
|---|---|---|
| `light` | 可停在 1 轮，但只适用于当前理解、关键假设和下一步建议 | 输出 `待确认`，不要扩写为完整方案 |
| `standard` | 对抽象输入或 B2B / 管理后台需求，通常至少 2 个逻辑轮次：先定目标/范围，再定规则/数据/体验中的高影响项 | 继续确认，或把关键缺口明确列为假设 |
| `strict` | 通常至少 3 个逻辑层：目标/范围、规则/数据、页面/生成输入；未得到用户确认时不能跳过 | 回退到继续确认，或等待授权假设 |

`standard` 模式下，若原始输入较抽象，结束确认前至少覆盖以下 4 类中的 3 类：

- 目标 / P0 场景 / 成功信号
- 角色 / 权限 / 使用边界
- 数据对象 / 字段粒度 / 生命周期
- 关键规则 / 冲突 / 异常 / 审计

`page-inventory` 或更高模式下，除上面内容外，还应覆盖：

- 页面或工作面选择
- 关键状态与入口/出口

B2B / 管理后台 / 配置治理需求，优先检查这组专用债务清单：

- 对象粒度是否明确，例如省级 / 市级 / 组织级
- 唯一性与冲突规则是否明确，例如单值、多值、覆盖、优先级
- 批量导入、批量操作或同步策略是否明确
- 角色权限和可见性是否明确
- 日志 / 审计 / 历史是否明确为 P0、P1 或假设
- 异常路径是否明确，例如校验失败、冲突、停用、删除、回滚

## questionDebt

内部维护，不需要每轮完整展示。用于判断是否还能结束反问。

- 目标 / 成功指标
- 用户 / 角色 / 权限
- P0 场景
- MVP 范围 / 非目标
- 核心流程
- 业务规则
- 数据对象 / 字段
- 状态机 / 生命周期
- 异常 / 审计 / 风险
- 页面清单 / 路由
- 交付模式 / 下游用途

每轮确认前，从 `questionDebt` 中选出影响最大的 1-3 个问题组。

每轮确认后，更新：

- `resolvedDebt`
- `remainingDebt`
- `assumptions`
- `nextAction`

仍有 `remainingDebt` 会影响范围、页面、业务规则、权限、数据、状态机、异常、验收或下游生成输入时，不得把需求标为已确认。

若 `remainingDebt` 中仍有 2 个以上高影响项属于以下类别，也不应结束确认：

- 对象 / 字段粒度
- 唯一性 / 冲突 / 覆盖策略
- 生命周期 / 状态 / 生效方式
- 权限 / 可见性 / 审计
- 批量导入 / 导出 / 同步 / 失败恢复

## confirmationCompleteness

对用户展示轻量版确认进度，避免大表格压迫感。

```markdown
### 确认进度

已确认：
- <2-4 条最关键已确认内容>

仍待确认：
- <2-4 条最高影响问题债务>

当前假设：
- <1-3 条当前使用的推荐假设>

下一步：
- `继续确认 <最高影响方向>`
- `接受当前假设，输出 <目标交付物>`
- `调整 <范围 / 页面 / 规则>`
```

展示原则：

- 每轮最多展示 2-4 条已确认、2-4 条待确认、1-3 条假设。
- 内部可以维护完整 `questionDebt`，但不要每轮把完整清单都贴给用户。
- 如果下一步是页面生成，下一步选项必须包含“继续确认”或“保留假设先生成”。
- 若宿主环境支持结构化选项或按钮，优先使用点击式选项；若不支持，使用 `A/B/C/D` 展示，并引导用户用 `都按推荐`、`A / B / A`、`第二题改 B` 这类更轻的方式回复。

### antiPrematureClosure

以下信号说明模型正在过早收口，应立即继续确认而不是补全细节：

- 两轮以内就开始输出完整方案，但 `remainingDebt` 仍覆盖对象、规则、权限、状态或页面中的多个核心项。
- 正文里连续出现多个“首版先按……假设”“默认按……设计”，且这些假设会改变字段、规则或页面。
- 用户明确指出“追问不够深”“别自己发挥”“先别出方案”“这块还没问到”。
- 用户刚推翻了已有页面或交互假设，例如把抽屉改成行内编辑，此时下游校验、保存策略、状态和边界也应重新确认。

出现这些信号时，应执行：

1. 提升到 `strict` 或保持当前更高深度
2. 重列最高影响 `remainingDebt`
3. 继续追问 1-3 个问题组
4. 暂停把结果包装成“完整方案”或“已确认页面清单”

## 选项质量

选项必须是产品决策包，不是单点偏好。

要求：

- 每个问题组提供 2-4 个互斥选项。
- 若支持点击式选项，优先用可点击标签；若不支持，则用 `A/B/C/D` 做展示标签，但不要要求用户回复 `1A/2B`。
- 选项覆盖当前决策空间的主要分支。
- 推荐选项放在第一位，并标注“推荐”。
- 每个选项说明对范围、页面、规则、数据、状态或验收中至少 2 项的影响。
- 不得提供只有标签、没有产品后果说明的选项。
- 如果选项不能覆盖主要分支，提供“其他/自定义”，并说明用户需要补充什么。

坏例子：

```markdown
1. 是否需要审批？
   - A. 需要
   - B. 不需要

问题：选项只是形式标签，没有说明产品后果；同时还要求用户回复 `1A`，交互成本偏高。
```

好例子：

```markdown
1. 首版流程深度选择？
   - A. 轻量登记流（推荐）：范围小；页面包含列表、详情、创建；不覆盖审批状态，验收聚焦基础 CRUD。
   - B. 标准审批流：增加提交、审核、驳回、通过；需要角色权限和状态机，验收覆盖审批路径。
   - C. 完整治理流：增加批量审核、审计记录、异常回退；页面和规则更多，适合正式 PRD。

请回复：
- `都按推荐`
- `A / B / A`
- `第二题改 B`
```

## 输出前门禁

输出产品方案、MVP 范围、PRD 方向、页面清单、页面级提示词或 HiUI 交接包前，必须检查确认完整度。

若仍有高影响 `remainingDebt`，只能三选一：

1. 继续确认
2. 标为假设
3. 等待用户明确授权假设

不得隐藏 `remainingDebt`，不得把带关键缺口的结果标为 `confirmed`。
